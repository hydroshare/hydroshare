# HydroShare Discovery Search

This document describes how the Atlas Search query in `discovery_atlas.py` is constructed,
what each clause type does, and how relevance scoring and boosts work.

Note: this documentation was generated by AI but reviewed and edited by a human.

---

## Overview

Search is powered by [MongoDB Atlas Search](https://www.mongodb.com/docs/atlas/atlas-search/),
which uses a Lucene-based engine with BM25 scoring. The main search query is a
[compound](https://www.mongodb.com/docs/atlas/atlas-search/compound/) query with four clause
types: `filter`, `must`, `should`, and (implicitly) `mustNot`. These are assembled in the
`SearchQuery.stages` property and sent as a MongoDB aggregation pipeline.

The index used is named `fuzzy_search`, defined in
`hs_discover/search_indexes/fuzzy_search.json`.

---

## Compound Clause Types

### `filter`
Hard constraints. A document **must** satisfy all filter clauses to appear in results.
Filter clauses **do not contribute to score** — they are pure inclusion/exclusion gates.

Used for:
- Excluding aggregation-type documents (those with no `dateCreated`)
- Date range constraints (`publishedStart`/`publishedEnd`, `dateCreatedStart`/`dateCreatedEnd`, etc.)
- Temporal coverage range filters
- Restricting to `type = ScientificDataset`

Because filter clauses are zero-cost for scoring, they are the right place for constraints
that users apply to narrow results but that shouldn't inflate relevance.

### `must`
A document **must** satisfy every `must` clause to appear in results. Unlike `filter`,
`must` clauses **do contribute to score** — the quality of the match affects ranking.

Used for:
- **Term search**: the free-text search term must match at least one indexed field
  (`name`, `description`, `keywords`, `creator.name`, `first_creator.name`, `contributor.name`)
- **Filter parameters** that require exact matches: `keyword`, `creatorName`, `providerName`,
  `contentType`, `creativeWorkStatus`, `hasPartName`, `isPartOfName`, etc.

When `term` is provided, it is placed inside a nested `compound.should` within `must`.
This means the term must match *at least one* of the six fields, but matching more fields
(or matching in higher-weighted fields) increases the score. See the
[Term Search and Scoring](#term-search-and-scoring) section for details.

### `should`
Optional clauses. Matching a `should` clause **increases the score** but is not required
for inclusion. The exception is when a compound has *only* `should` clauses and no
`filter` or `must` — in that case at least one `should` must match. This applies to the
`typeahead` endpoint, which uses a `should`-only compound. The main search compound always
has `filter` clauses present, so `should` is genuinely optional there.

Used for:
- **Phrase boost**: if the term is multi-word, bonus score for documents where the words
  appear adjacent (exact phrase), rather than just anywhere in the field
- **Filter-specific autocomplete boosts**: when `creatorName`, `keyword`, or
  `fundingFunderName` filters are active, an additional autocomplete clause fires to reward
  documents that closely match the filter value beyond the hard `text` match in `must`
- **Published resource boost**: a `range` clause on `datePublished` that fires for all
  published resources, giving them a moderate ranking advantage over unpublished ones.

## Atlas Search Operators

### `autocomplete`
Matches tokens using an **edge n-gram index** — the field is pre-indexed as prefixes of
each word. A query of `"wat"` matches any token starting with `"wat"` (water, watershed, etc.).

For multi-word queries like `"water level"`, Atlas tokenizes them and matches each word
**independently**. A document containing "water" anywhere matches the "water" token; a
document containing "level" anywhere matches the "level" token. Documents with both score
higher than documents with only one.

**Important**: autocomplete scores using BM25, which means:
- Common words (appearing in many documents) get lower IDF → lower base score
- Rare/specific words get higher IDF → higher base score
- Short documents with the term appearing frequently score higher than long documents
  with the same number of occurrences (field length normalization)

### `text`
Standard tokenized full-text match. Unlike autocomplete, it only matches whole tokens
(words), not prefixes. It uses standard word boundaries — notably, **underscores are not
word separators**, so `"ciroh_portal"` is treated as a single token. Used in `must` for
filter parameters (`keyword`, `creatorName`, etc.) because it requires an exact word match.

### `phrase`
Matches the query words in order, adjacent to each other. Used in `should` to give a
score bonus to documents where the multi-word term appears as a phrase rather than just
having the words scattered throughout.

Example: for query `"water level"`, a document titled `"Water Level Monitoring Station"`
gets the phrase boost; a document titled `"Water Quality at High Level Sites"` does not
(words are not adjacent).

### `range`
Binary match: a date field is either within the range or it isn't. Range clauses in
`filter` just gate inclusion and don't affect scoring. The range clause in `should` for `datePublished`
is the exception — it contributes a flat score bonus for any published resources.

**Important scoring note**: unlike BM25 operators, a `range` operator always has a base
score of 1.0 (it either matches or it doesn't; there is no "how well" for a date). This
means `score: {boost: {value: N}}` on a range clause always adds exactly `N` to the
document's score — a flat constant, not a variable multiplier.

### `term`
Exact value match with no analysis or tokenization. Used for `contentType` and
`creativeWorkStatus` to ensure precise categorical filtering.

Note: this is the Atlas Search `term` *operator*, not to be confused with the `term`
*query parameter* in the search API, which is the free-text search input.

## Term Search and Scoring

When a user provides a `term` (free-text search), the query is structured as a nested
compound inside `must`:

```
must:
  compound:
    should:
      - autocomplete(term, 'name',              boost=5)
      - autocomplete(term, 'description',       boost=3)
      - autocomplete(term, 'keywords',          boost=3)
      - autocomplete(term, 'creator.name',      boost=5)
      - autocomplete(term, 'first_creator.name',boost=5)
      - autocomplete(term, 'contributor.name',  boost=5)
```

Note: Scoring boosts are configured in `hydroshare/settings.py` — see the [Settings Reference](#settings-reference) for current values.

A document must match at least one of these to appear in results at all. Every clause
that matches contributes its score additively to the document's total.

**Scores accumulate across fields.** Example for query `"water quality"`:

| Field          | BM25 base | Boost | Contribution |
|----------------|-----------|-------|-------------|
| name           | 2.5       | ×5    | 12.5        |
| description    | 1.0       | ×3    | 3.0         |
| keywords       | 1.5       | ×3    | 4.5         |
| creator.name   | 0 (miss)  | ×5    | 0           |
| **Total**      |           |       | **20.0**    |

A document with the term in multiple fields outscores one with the term in only one field.

### Why `name` and creator fields have higher boosts

The title of a resource (`name`) is the strongest relevance signal — if "water quality"
is in the title, the resource is almost certainly about water quality. A match in
`description` is weaker signal because descriptions can be long and mention many tangential
topics. Creator name matches are given the same weight as title matches because if a user
searches for a person's name and finds it in the creator field, that is a precise match.

### Fuzzy matching

For search terms of 10 or more characters (`SEARCH_FUZZY_MIN_TERM_LENGTH`), the autocomplete
query enables fuzzy matching with `maxEdits: 1`. This allows one character insertion,
deletion, or substitution, helping with typos. It is disabled for short terms to avoid
spurious matches.

### Phrase boost

For multi-word terms, additional `phrase` clauses fire in `should`:

```
should:
  - phrase(term, 'name',        boost=15)
  - phrase(term, 'description', boost=9)
  - phrase(term, 'keywords',    boost=9)
```

These do not affect inclusion (they are in `should`, not `must`). They reward documents
where the words appear adjacent. A document titled `"Water Level Forecast"` gets a score
boost on top of its autocomplete score when the term is `"water level"`.

The phrase boosts are intentionally higher than the autocomplete boosts so that an exact
phrase match clearly outranks a scattered-word match.

## Published Resource Boost

Published resources (those with a non-null `datePublished`) receive a ranking bonus via
a `range` clause in `should`:

```
should:
  - range(datePublished >= datetime.min, boost=10)
```

Because range operators have a base score of 1.0, this always adds exactly **+10** to
a published document's score, regardless of text relevance.

**Why this is in `should` and not `must`**: unpublished resources should still appear in
results — publication status is a ranking signal, not an eligibility requirement.

**Why this cannot cause irrelevant published documents to surface**: the term search
clauses are in `must`. A document must match the term in at least one field before it
reaches scoring at all. The `+10` from `datePublished` only affects the rank among
documents that already matched the term.

**Tiebreaker behavior**: +10 is calibrated to be a meaningful-but-not-dominant boost.
Typical BM25 scores for a genuine text match range from ~0.5 to ~20+. So for a term search on
"water quality":

| Document                                                | Text score | Published bonus | Total    | Rank |
|---------------------------------------------------------|------------|-----------------|----------|------|
| Published: *"Water Quality Dataset"* (strong)           | 10         | +10             | **24**   | 1st  |
| Unpublished: *"Water Quality Study"* (strong)           | 10         | 0               | **14**   | 2nd  |
| Published: *"Qualification of Monitoring"* (weak)       | 3          | +10             | **13**   | 3rd  |
| Unpublished: *"Quality of Water Monitoring"* (moderate) | 7           | 0              | **7**    | 4th  |

A published document with a weak match can still outrank an unpublished document with a
moderate match. If pure relevance ordering is preferred, the boost value can be reduced
in `settings.SEARCH_BOOST_DATE_PUBLISHED`.

## Relevance Threshold

When `term`, `creatorName`, or `keyword` is provided, a post-search filter removes
documents scoring below `SEARCH_RELEVANCE_SCORE_THRESHOLD` (current: `0.3`):

```python
{'$match': {'score': {'$gt': settings.SEARCH_RELEVANCE_SCORE_THRESHOLD}}}
```

This is a safety net against extremely weak matches that technically satisfied the `must`
clause but have essentially no relevance to the query. In practice, with the `must` clause
gating on term presence, most genuine matches score well above 0.3; this threshold mainly
catches edge cases like a very common word appearing once in an extremely long description.

**Note on score stability**: BM25 IDF is computed across the entire collection. As the
number of resources grows and common terms appear in more documents, their IDF decreases
and scores for those terms drift downward. The threshold may need periodic recalibration
as the corpus grows.

## Highlights

The response includes highlight snippets from matched fields:

```python
highlightPaths = ['name', 'description', 'keywords', 'creator.name', 'contributor.name']
```

**Highlights are independent from scoring.** The highlight score attached to each snippet
is Atlas Search's assessment of how informative that particular passage is for display
purposes — it is not the clause's contribution to the document's overall relevance score.
A document can score highly with an unimpressive highlight (e.g., a common word appearing
many times in a long description), or show a clear highlight snippet while scoring
relatively low overall.

---

## Pagination

Pagination uses Atlas Search's `searchAfter` cursor-based approach via `paginationToken`
(a `searchSequenceToken` from the previous response). This is more efficient than
`$skip`-based pagination for deep pages and maintains consistent ordering even if
documents are modified between requests.

When no `paginationToken` is provided, `$skip` offset pagination is used as a fallback
(controlled by `pageNumber` and `pageSize`).

## Settings Reference

All boost values and thresholds are defined in `hydroshare/settings.py`:

| Setting                                   | Purpose |
|-------------------------------------------|---------|
| `SEARCH_RELEVANCE_SCORE_THRESHOLD`        | Minimum score to appear in term/filter search results |
| `SEARCH_BOOST_NAME`                       | Autocomplete boost for term match in resource title |
| `SEARCH_BOOST_PHRASE_NAME`                | Phrase boost for exact phrase match in title |
| `SEARCH_BOOST_DESCRIPTION`                | Autocomplete boost for term match in description |
| `SEARCH_BOOST_PHRASE_DESCRIPTION`         | Phrase boost for exact phrase match in description |
| `SEARCH_BOOST_KEYWORDS`                   | Autocomplete boost for term match in keywords |
| `SEARCH_BOOST_PHRASE_KEYWORDS`            | Phrase boost for exact phrase match in keywords |
| `SEARCH_BOOST_CREATOR_NAME`               | Autocomplete boost for term match in creator name |
| `SEARCH_BOOST_FIRST_CREATOR_NAME`         | Autocomplete boost for term match in first creator name |
| `SEARCH_BOOST_CONTRIBUTOR_NAME`           | Autocomplete boost for term match in contributor name |
| `SEARCH_BOOST_CREATOR_NAME_FILTER`        | Additional autocomplete boost when `creatorName` filter is active |
| `SEARCH_BOOST_FIRST_CREATOR_NAME_FILTER`  | Additional autocomplete boost for first creator when `creatorName` filter is active |
| `SEARCH_BOOST_CONTRIBUTOR_NAME_FILTER`    | Additional autocomplete boost for contributor when `creatorName` filter is active |
| `SEARCH_BOOST_KEYWORD_FILTER`             | Additional autocomplete boost when `keyword` filter is active |
| `SEARCH_BOOST_FUNDING_FUNDER_NAME_FILTER` | Additional autocomplete boost when `fundingFunderName` filter is active |
| `SEARCH_BOOST_DATE_PUBLISHED`             | Flat score bonus for published resources |
| `SEARCH_FUZZY_MIN_TERM_LENGTH`            | Minimum term length (characters) to enable fuzzy matching |
